# Техническое задание: модуль просмотра и ревью файлов Lore

## 1. Назначение

Разработать read-only модуль `/files` для просмотра результатов обработки файлов системой Splitter и оставления локальных комментариев.

Пользователь должен иметь возможность:

- найти нужный файл;
- посмотреть текущую и предыдущие обработки файла;
- изучить разбиение файла на чанки;
- сравнить разные текстовые представления чанка;
- просмотреть связанные таблицы, изображения и расшифровки;
- увидеть ошибки, предупреждения и результаты автоматической проверки;
- оставить комментарий или отметить результат как корректный;
- экспортировать и импортировать результаты ревью.

## 2. Формат приложения

- React + TypeScript SPA.
- Роут: `/files`.
- Модуль должен быть встроен в навигацию Lore.
- API вызывается через same-origin.
- Не реализовывать модуль как большой кастомный компонент Chainlit.
- Интерфейс полностью read-only по отношению к серверным данным.

Пользователь не может:

- редактировать файл, чанк, таблицу или изображение;
- изменять результаты обработки и диагностики;
- перезапускать обработку или аудит;
- сохранять комментарии на сервере;
- выполнять произвольные SQL-запросы.

## 3. Основной пользовательский сценарий

1. Пользователь открывает `/files`.
2. Находит файл через поиск или фильтры.
3. Выбирает файл — по умолчанию открывается его последняя успешная обработка.
4. При необходимости выбирает исторический запуск.
5. Последовательно просматривает чанки файла.
6. Сравнивает `Display`, `Fulltext` и `Vectortext`.
7. Проверяет координаты, метаданные, диагностику и связанные payload.
8. Просматривает таблицы, изображения или расшифровку.
9. Оставляет оценку `OK`, `Проблема` или `Вопрос`.
10. Просматривает все локальные комментарии и экспортирует отчёт.
11. При необходимости импортирует JSON-отчёт автоматического AI-аудита.

## 4. Основной экран

Интерфейс состоит из трёх панелей.

### 4.1. Левая панель — список файлов

Для каждого файла отображать:

- название и тип;
- статус обработки;
- время последнего запуска;
- количество чанков;
- количество таблиц и изображений;
- количество диагностик;
- количество локальных комментариев;
- количество непросмотренных автоматических замечаний.

Поддержать поиск и фильтры:

- название файла;
- тип файла;
- pipeline;
- статус: `success`, `active`, `failed`, `skipped`;
- период обработки;
- наличие warning/error;
- наличие таблиц;
- наличие изображений;
- наличие комментариев;
- наличие автоматических замечаний;
- аномальное количество чанков.

Сортировки:

- по названию;
- времени обработки;
- статусу;
- количеству проблем;
- количеству чанков.

Список должен использовать серверную пагинацию и корректно отображать продолжение выборки.

### 4.2. Центральная панель — структура файла

В шапке файла отображать:

- название и тип файла;
- выбранный запуск;
- селектор исторических запусков;
- статус;
- количество чанков и payload;
- количество ошибок и предупреждений;
- статус автоматической проверки;
- ссылку на текущий исходный объект;
- ссылку на внешний оригинал, если она доступна;
- действие «Сравнить запуски».

Ниже показывать чанки в детерминированном порядке по `ordinal`.

Карточка чанка должна содержать:

- `ordinal` и тип;
- координаты в исходном документе;
- короткий preview `display_text`;
- количество символов и токенов;
- типы связанных payload;
- предупреждения и автоматические замечания;
- состояние локального ревью.

Фильтры чанков:

- тип чанка;
- тип payload;
- наличие диагностики или замечания;
- состояние ревью;
- координаты или раздел документа.

Поддержать переход к предыдущему и следующему чанку с клавиатуры.

### 4.3. Правая панель — инспектор чанка

Вкладки:

- `Display`;
- `Fulltext`;
- `Vectortext`;
- `Payloads`;
- `Metadata`;
- `Diagnostics`.

Для текстовых представлений поддержать:

- отрендеренный Markdown;
- точный raw-режим;
- копирование текста;
- количество символов и токенов;
- хеш текста;
- координаты;
- breadcrumbs;
- просмотр соседних чанков;
- выделение фрагмента для комментария.

Добавить diff по запросу пользователя:

- `Fulltext ↔ Vectortext`;
- `Display ↔ Fulltext`;
- side-by-side режим;
- unified режим.

Допускается режим визуальной нормализации пробелов, но точная сохранённая строка всегда должна оставаться доступной.

Все три панели должны быть сворачиваемыми и изменяемыми по ширине.

## 5. Deep links

В URL необходимо сохранять:

- выбранный файл;
- запуск;
- чанк;
- вкладку инспектора;
- активные фильтры;
- параметры сравнения запусков.

После перезагрузки страницы состояние должно восстанавливаться из URL.

Локальные комментарии не входят в deep link и не передаются другому пользователю вместе со ссылкой.

## 6. Просмотр таблиц

Для табличного payload отображать:

- сохранённое summary;
- координаты в источнике;
- схему и типы колонок;
- количество строк и столбцов;
- статистику и сохранённые samples;
- идентификатор содержимого;
- список мест использования;
- связанные чанки.

Табличный просмотр должен поддерживать:

- серверную пагинацию;
- виртуальный скролл;
- сортировку колонок;
- фильтры по колонкам;
- глобальный поиск значения;
- изменение ширины колонок;
- скрытие и закрепление колонок;
- явное отображение `NULL` и типа значения;
- копирование ячейки, строки или выбранного диапазона;
- экспорт текущей отфильтрованной выборки в CSV;
- переход из чанка к таблице и обратно.

Можно показывать read-only preview запроса, полученный от сервера. SQL-редактора быть не должно.

Не входят в задачу:

- редактирование данных;
- формулы;
- графики;
- BI-конструктор.

## 7. Просмотр изображений

Для изображения отображать:

- пропорциональный preview;
- полноразмерный просмотр;
- сохранённое описание;
- MIME-тип;
- размеры;
- размер файла;
- content hash;
- координаты в источнике;
- классификацию;
- предупреждения;
- список мест использования;
- связанные чанки;
- временную ссылку на объект.

Если изображение недоступно, интерфейс должен продолжать показывать его описание и метаданные с явным состоянием «Объект недоступен».

Фронтенд не должен перегенерировать описание или менять классификацию изображения.

## 8. Просмотр расшифровок

Для транскриптов поддержать:

- группировку по сохранённым блокам или темам;
- фильтр по спикеру;
- навигацию по спикерам;
- отдельное отображение известных, неизвестных и неоднозначных спикеров;
- навигацию по временным интервалам;
- переход к следующей реплике выбранного спикера;
- timeline с границами чанков;
- индикацию интервалов, которые попали в несколько чанков;
- стандартный diff текстовых представлений;
- создание комментария к выделенной цитате.

Назначение текстовых вкладок:

- `Display` — спикеры, таймкоды и форматированное представление;
- `Fulltext` — исходная формулировка для поиска;
- `Vectortext` — очищенный самодостаточный текст.

## 9. Сравнение запусков

Пользователь выбирает два запуска одного файла.

Показать:

- добавленные чанки;
- удалённые чанки;
- изменённые чанки;
- неизменившиеся чанки;
- изменения текстовых хешей;
- изменения связанных payload;
- изменения диагностик.

Сопоставлять чанки по `content_signature`, а не только по `ordinal`.

Оба `run_id` должны сохраняться в URL.

Если локальный комментарий относится к совпадающему `content_signature`, интерфейс может предложить перенести его на новый запуск, но не должен делать это автоматически.

Поскольку исторические версии исходного файла могут отсутствовать, при сравнении необходимо показывать предупреждение о версии источника.

## 10. Локальные комментарии и ревью

При первом открытии пользователь должен указать имя ревьюера. Имя можно изменить позднее.

Для каждого комментария сохранять:

- вердикт: `OK`, `problem` или `question`;
- одну или несколько категорий;
- текст комментария;
- опциональную выделенную цитату;
- имя ревьюера;
- время создания и изменения;
- состояние: `open` или `reviewed`;
- окружение/base URL;
- ключ файла;
- `run_id`;
- тип и ID объекта;
- `content_signature`;
- `ordinal` и координаты, если применимо;
- источник: `human` или импортированный `agent`.

Поддерживаемые категории:

- граница чанка;
- потеря контекста;
- структура или заголовки;
- `Display`;
- `Fulltext`;
- `Vectortext`;
- размер чанка;
- дублирование;
- таблица;
- изображение;
- summary;
- ссылка на payload;
- спикер;
- таймкод;
- ссылка на источник;
- диагностика;
- другое.

Панель комментариев должна поддерживать:

- создание;
- редактирование;
- удаление;
- группировку;
- фильтрацию;
- перевод между `open` и `reviewed`;
- копирование в буфер;
- возврат к объекту через deep link;
- отображение состояния «Объект больше не найден»;
- полную очистку локальных данных;
- экспорт и импорт для одного файла, одного запуска или всех файлов.

## 11. Локальное хранение

В IndexedDB хранить:

- комментарии;
- импортированные AI-замечания;
- имя ревьюера;
- статусы проверки;
- отметки false positive;
- черновики.

`localStorage` использовать только для небольших UI-настроек: размеры панелей, выбранная тема и аналогичные предпочтения.

Серверная синхронизация комментариев не предусматривается.

## 12. Импорт и экспорт

Поддержать два формата:

- Markdown — читаемый отчёт для передачи другому человеку;
- версионированный JSON — полный переносимый формат с возможностью обратного импорта.

Требования:

- повторный импорт не создаёт дубликаты: дедупликация по ID комментария или finding;
- JSON валидируется по версии схемы;
- должны быть ограничения на размер импортируемого файла;
- полные тексты чанков и полные таблицы в экспорт не включаются;
- экспорт содержит только выбранные цитаты, технический контекст и ссылки на объекты;
- импортированный Markdown и текст отображаются без исполнения HTML или JavaScript;
- в одном итоговом отчёте можно объединить человеческие комментарии и импортированные AI-замечания.

## 13. Состояния и ошибки

Обязательные состояния:

- загрузка;
- пустой список;
- файл или запуск не найден;
- активная обработка;
- успешная обработка;
- `failed`;
- `skipped`;
- частично доступные данные;
- источник недоступен;
- версия текущего источника не совпадает с выбранным запуском;
- таблица или изображение недоступны;
- ответ обрезан серверным лимитом;
- timeout;
- некорректный cursor.

Одна неработающая панель не должна блокировать уже загруженные данные в остальных панелях.

`failed` и `skipped` нельзя визуально представлять как успешный запуск с пустым списком чанков.

При несовпадении версии источника необходимо явно предупредить пользователя и не выдавать текущий источник за версию, соответствующую историческому запуску.

## 14. Безопасность

- Не исполнять HTML и JavaScript из Markdown, комментариев, диагностик и импортированных файлов.
- Не логировать в браузерную аналитику содержимое чанков, таблиц и изображений.
- Не сохранять временные ссылки на объекты дольше срока их действия.
- Не показывать пользователю DSN, credentials, connection payload и другие внутренние данные.
- Не формировать произвольные SQL-запросы на клиенте.
- Все данные, полученные от API, считать недоверенным пользовательским контентом.

## 15. Адаптивность и доступность

- Все основные операции должны быть доступны с клавиатуры.
- Должен быть видимый focus state.
- Вкладки, панели, таблицы, диалоги и фильтры должны иметь корректную семантику и ARIA-разметку.
- Цвет не должен быть единственным способом обозначения статуса или severity.
- Интерфейс должен поддерживать уменьшенную ширину экрана; для узких экранов допускается последовательное отображение панелей вместо одновременного.
- Изменение размера и сворачивание панелей не должно делать управление недоступным с клавиатуры.

## 16. Ожидаемый read-only API

Фронтенд не должен самостоятельно конструировать доступ к хранилищам. Все данные получаются через версионированный HTTP API.

Основные операции:

- список, поиск и фильтрация файлов;
- карточка файла и история запусков;
- карточка запуска и его manifest;
- список и карточка чанков, соседние чанки;
- сравнение двух запусков;
- metadata, profile, sample и строки табличного payload;
- preview и содержимое изображения;
- диагностики и разрешение ссылок;
- ограниченный контекст текущего источника.

Клиент должен обрабатывать машинные коды ошибок `NOT_FOUND`, `NOT_AVAILABLE`, `SOURCE_VERSION_MISMATCH`, `PAYLOAD_NOT_REGISTERED`, `TOO_LARGE`, `TRUNCATED`, `UPSTREAM_TIMEOUT` и `INVALID_CURSOR`.

Ответы могут содержать `truncated` и continuation cursor. Фронтенд обязан показывать ограниченность данных и не интерпретировать обрезанный ответ как полный.

## 17. Критерии приёмки

- Реализован React/TypeScript-модуль `/files` внутри навигации Lore.
- Работает трёхпанельный просмотр файлов, запусков и чанков.
- Доступны три текстовых представления, raw/rendered режимы и diff.
- Работает просмотр таблиц, изображений и транскриптов.
- Работает сравнение двух запусков.
- Комментарии сохраняются локально в IndexedDB и не отправляются на сервер.
- Работают редактирование, удаление, фильтрация и статусы комментариев.
- Работает безопасный импорт/экспорт Markdown и версионированного JSON.
- URL восстанавливает выбранный файл, запуск, чанк, вкладку и фильтры.
- Корректно отображаются `success`, `active`, `failed`, `skipped` и degraded-состояния.
- Отсутствие изображения, таблицы или источника не блокирует просмотр остальных данных.
- Реализована клавиатурная навигация и базовая accessibility-проверка.
- Компонентными тестами покрыты инспектор, diff, таблица, транскрипты, IndexedDB и импорт.
- Browser E2E покрывают как минимум:
  1. документ с таблицей и изображением;
  2. workbook с большой таблицей;
  3. транскрипт с несколькими спикерами;
  4. `failed`/`skipped` запуск с диагностикой;
  5. импорт AI-отчёта и совместный экспорт AI-замечаний и человеческих комментариев.

## 18. Границы HTML-прототипа

Пример интерфейса поставляется отдельным автономным HTML-файлом без сборки и внешних зависимостей. Он должен демонстрировать выбранное направление C — сбалансированный трёхпанельный инспектор — и содержать:

- демонстрационные файлы и чанки;
- поиск и переключение выбранного файла;
- рабочие вкладки инспектора;
- raw/rendered режимы;
- diff текстовых представлений;
- табличный и графический payload;
- создание локального комментария;
- список комментариев;
- имитацию импорта/экспорта;
- переключение normal/degraded состояния;
- адаптивное представление.

Прототип не является реализацией React-приложения, не вызывает настоящий API и не определяет окончательную структуру production-компонентов.
