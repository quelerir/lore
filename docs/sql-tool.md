# SQL-инструмент (toast): что решает и как устроен

Полное описание инструмента ответов на вопросы по toast-таблицам:
`backend/toast/` (граф, guardrails, исполнитель) + обвязки (Studio, eval,
демо-режим). Документ отвечает на три вопроса: какую задачу решаем, как
устроен каждый узел и почему именно так.

## 1. Решаемая задача

В Postgres (внешняя БД `loreagent_test`, схема `splitter_toast`) лежат
таблицы, извлечённые сплиттером из XLSX-документов: реестры сотрудников,
матрицы грейдов, справочники. Пользователь задаёт вопрос на естественном
языке («Какие ФИО у юристов и их должности?»), система должна ответить
строго по данным таблицы.

Особенности данных, которые определили дизайн:

- **Физические имена колонок непрозрачны.** Сплиттер называет колонки
  `column_1, column_2, …` либо выводит имена из строки-заголовка, которая
  может оказаться строкой данных (`senior_legal_manager`,
  `j_suvorova_adventum_ru` — слаги из чужой строки). Смысл колонок знает
  только человек.
- **Смысл задаётся рукописным описанием.** У каждой таблицы есть два
  описания: `desc_vector` (краткое, для векторного поиска) и `desc_full`
  (полное, с маппингом «колонка → смысл»). `desc_full` пишется человеком и
  передаётся не через LLM — считается авторитетным источником имён и смысла
  колонок.
- **Вопросы требуют вычислений.** «Сколько человек в реестре», «у кого
  грейд выше» — это фильтры и агрегации. Складывать таблицу в текстовый
  RAG-чанк бессмысленно: нужен SQL.

Отсюда форма решения: **LLM генерирует SQL по описанию таблицы, детермини-
рованный исполнитель его выполняет, LLM суммирует результат.** Инструмент
работает над ОДНОЙ таблицей за вызов — выбор таблицы под вопрос (retrieval)
делает внешний пайплайн, который разрабатывается отдельно.

## 2. Жёсткие ограничения среды

Эти пункты не обсуждаются при доработках:

1. **Toast-БД неприкосновенна.** Никакого DDL, ролей, GRANT — БД внешняя.
   Вся защита живёт на стороне приложения (см. раздел 7).
2. **Юзер БД — пишущий.** Read-only приходится навязывать самим, на каждую
   транзакцию.
3. **Перед БД стоит transaction-pooling пулер (PgBouncer).** Он рвёт
   соединения со startup-параметрами (`unsupported startup parameter`),
   поэтому `default_transaction_read_only` и `statement_timeout` нельзя
   задать на уровне соединения — только `SET LOCAL` внутри транзакции.
4. **Модели — через OpenRouter** (единый ключ, разные модели). Поддержка
   function calling у моделей неровная — отсюда фолбэки.

## 3. Контракт инструмента

Вход (`SqlToolInput`):

| Поле | Что это |
|---|---|
| `question` | вопрос пользователя на естественном языке |
| `chunk_id` | id чанка-источника (прокидывается в ответ как есть) |
| `table` | физическое имя таблицы `toast_tbl_<20 hex>` |
| `desc_vector` | краткое описание таблицы |
| `desc_full` | полное рукописное описание с маппингом колонок |

Выход (`run_sql_tool` → `_project`):

```json
{
  "status": "ok | no_data | error",
  "answer": "текст ответа",
  "chunk_id": "...", "table": "...",
  "sql_attempts": [{"sql", "ok", "error", "row_count"}, ...],
  "rows_used": 3
}
```

`status` различает три исхода: ответ по данным (`ok`), «SQL выполнился, но
данных под вопрос нет» (`no_data`), техническая невозможность (`error`).
`sql_attempts` — полная история попыток для отладки и UI.

## 4. Почему langgraph-граф, а не «агент с инструментами»

Дисциплина шагов и бюджет **зашиты в код** и не зависят от качества модели:
LLM вызывается ровно в трёх местах (generate/judge/summarize), всё
остальное — детерминированные узлы и маршрутизация. Слабая модель может
сгенерировать плохой SQL, но не может «забыть» проверить результат, уйти в
бесконечный цикл или потратить лишние запросы. Это осознанный выбор против
ReAct-агента, где порядок действий определяет сама модель.

Топология:

```
START → init → generate → execute(∥) → judge → summarize → END

условные переходы:
  generate → summarize   если модель не дала ни одного кандидата
  execute  → summarize   если бюджет запросов исчерпан (минуя судью)
  judge    → generate    если строк недостаточно (ещё раунд)
```

## 5. Узлы: что делает и почему именно так

### init — детерминированный старт без БД

Инициализирует аккумуляторы состояния (`attempts=[]`, `executed_count=0`,
`round=0`). Явный узел вместо `state.get(...)` по всему коду — одна точка,
где гарантируется наличие ключей.

**Почему без похода в БД.** Ранняя версия имела узел `scope`, тянувший
реальные колонки из `information_schema`. От него отказались: `desc_full`
рукописный и авторитетный, реальные имена без смысла беднее описания.
Остаточный риск (физическую колонку переименовали, описание не обновили)
закрыт бэкстопом: SQL упадёт на выполнении, Postgres-ошибка уйдёт в
retry-цикл generate.

### generate — LLM выдаёт батч SQL-кандидатов

На вход: вопрос, имя таблицы, оба описания, число кандидатов `n` (минимум
из `candidates_per_round` и остатка бюджета) и фидбек прошлых раундов.

- **Батч РАЗНЫХ кандидатов, а не один запрос.** Кандидаты выполняются
  параллельно; разные подходы к одному вопросу повышают шанс попасть в
  нужные колонки при непрозрачной схеме.
- **Structured output с фолбэком.** Сначала
  `model.with_structured_output(SqlCandidates, method="function_calling")`;
  при любом исключении (модель без function calling, фейк в тестах) —
  обычный `ainvoke` + текстовый парсер `parse_sql_candidates` (JSON-массив
  или строки, начинающиеся с SELECT). Парсер не удалён именно потому, что
  он и есть фолбэк.
- **Фидбек между раундами.** В промпт попадают последние ошибки SQL
  («исправь») и запросы, вернувшие 0 строк («нужен другой подход») — иначе
  модель повторяет один и тот же пустой запрос и жжёт бюджет.
- **Тег `internal`.** Все служебные LLM-вызовы (generate, judge) помечены
  тегом — UI-слой по нему отфильтровывает их токены из стрима ответа.

### execute — параллельное выполнение кандидатов

Гоняет кандидатов раунда через `asyncio.gather` по read-only исполнителю.

- **`return_exceptions=True`** — сбой одного кандидата (сеть, DNS) не
  роняет остальных и весь граф: исключение конвертируется в неуспешную
  попытку и дополнительно пишется в лог (иначе инфраструктурные проблемы
  видны только в UI).
- **Дедупликация с учётом бюджета.** SQL, уже выполнявшийся в прошлых
  раундах, повторно не гоняется, но бюджет списывается за весь батч.
  Тонкость: если списывать только за выполненные, раунд из одних повторов
  не двигал бы счётчик — и цикл не завершался бы.
- Каждая попытка — типизированная запись `Attempt` (`sql, ok, error, rows,
  row_count, truncated`), накапливается в состоянии.

### judge — LLM-проверка достаточности

Отвечает на вопрос «достаточно ли полученных строк, чтобы ответить?» —
ловит случай «строки есть, но не по теме» (SELECT вернул не те колонки или
не тот срез).

- **Не зовёт модель, если строк нет** — вердикт `need_more` детерминирован
  (экономия вызова).
- **Парсинг вердикта устойчив к болтливости**: `sufficient` матчится по
  границе слова и отклоняется при наличии `need_more` в ответе — «Данных
  мало: INSUFFICIENT» не превращается в согласие (регресс-тест).
- **Кап контекста двойной**: максимум `JUDGE_ROWS_CAP=30` строк И
  ~`JUDGE_CONTEXT_CHARS=8000` символов сериализованного JSON — одна
  «широкая» строка не раздувает промпт. Судье честно сообщается «Показано
  строк: X из Y».

### summarize — терминальный узел

Три ветки:

1. Есть строки → LLM формулирует ответ СТРОГО по ним (промпт запрещает
   выдумывать), `status=ok`. Единственный незатегированный вызов — его
   токены стримятся пользователю.
2. Строк нет, но был хоть один успешный SELECT → `status=no_data`,
   фиксированный текст «В данных таблицы нет ответа на этот вопрос».
3. Все попытки — ошибки (или кандидатов не было вовсе) → `status=error`
   с текстом последней ошибки.

Разделение `no_data`/`error` принципиально: «данных нет» — валидный ответ
системы, «не смогли выполнить SQL» — технический сбой.

## 6. Управление циклом и гарантия завершаемости

Бюджет `max_queries` — жёсткий предел ЧИСЛА выполненных SQL за все раунды,
проверяется в `after_execute` ДО судьи (не тратим вызов судьи, если
продолжать всё равно нельзя). Завершаемость доказывается просто: каждый
проход цикла либо списывает ≥1 из конечного бюджета (батч непуст), либо
выходит в summarize (батч пуст — ветка `after_generate`). Ветка пустого
батча появилась после инцидента: модель, отвечающая текстом без SELECT,
крутила граф до `GraphRecursionError`.

## 7. Безопасность: три слоя на стороне приложения

БД менять нельзя (раздел 2), поэтому защита целиком в коде. Слои
независимы — пробитие одного не открывает данные.

### Слой 1: AST-guardrails (`sql_guardrails.py`)

Валидация по AST через **sqlglot** (диалект postgres), не по regex:

- ровно одна команда; только `SELECT` (включая UNION/INTERSECT/EXCEPT);
- `SELECT INTO` и `FOR UPDATE/SHARE` — отказ (это запись/блокировки);
- **все** упомянутые таблицы обязаны быть `splitter_toast.<переданная>`:
  проверяются FROM, JOIN, **comma-join** и подзапросы в любом месте;
- CTE разрешены: алиасы `WITH` исключаются из проверки таблиц, но таблицы
  внутри CTE проверяются как обычно; ссылка без схемы на не-алиас — отказ;
- табличные функции (`generate_series(...)`) — отказ (fail-closed);
- строковые литералы ложных отказов не дают: `WHERE x = 'create; drop'`
  проходит — это данные, а не команды.

История: первая версия была на regex и пропускала comma-join
(`FROM splitter_toast.t, public.users`) — чтение чужих таблиц. Ровно
поэтому валидация переписана на AST; regex к SQL принципиально дыряв.

Отказ возвращается СТРОКОЙ с префиксом «Отказ: …» — он попадает в фидбек
generate, и модель переписывает запрос.

### Слой 2: read-only транзакция (`executor.py`)

Каждый запрос выполняется как `BEGIN READ ONLY` + `SET LOCAL
statement_timeout = 5000` — по-транзакционно, потому что пулер режет
startup-параметры (раздел 2). Даже если guardrails пропустят мутацию,
Postgres её отвергнет.

### Слой 3: лимиты исполнителя

- **LIMIT на стороне БД**: запрос оборачивается в
  `SELECT * FROM (…) AS _q LIMIT 201` (`MAX_ROWS=200` + 1 строка для флага
  `truncated`) — миллионы строк не едут в память приложения;
- серверный `statement_timeout` (5 с) + клиентский `command_timeout`
  asyncpg; клиентский `TimeoutError` ловится и возвращается строкой ошибки,
  а не роняет граф;
- значения строк приводятся к JSON-совместимым типам (`_plain`): bytea
  декодируется с заменой битых байтов, Decimal/даты/UUID — в строку.

Исполнитель держит **соединение на запрос** (не пул): перед БД и так пулер,
а одно asyncpg-соединение нельзя делить между параллельными кандидатами.

## 8. Модель

Отдельная «умная» модель `SQL_MODEL` (по умолчанию
`anthropic/claude-sonnet-4.6`) через OpenRouter, temperature 0 — она
дороже чат-модели, но SQL-генерация по непрозрачной схеме того требует.

Важный нюанс OpenRouter: **предел вывода надо передавать родным полем
`max_tokens` через `extra_body`** — langchain-openai шлёт
`max_completion_tokens`, который OpenRouter игнорирует и резервирует полное
окно модели (65 536 токенов у sonnet). При малом балансе это даёт 402 ещё
до генерации. Управляется env `LLM_MAX_TOKENS`.

## 9. Конфигурация

Все параметры — в `backend/config.py` (pydantic-settings), env один на весь
стек (корневой `.env`: его читают и docker-compose, и LangGraph Studio).
Пустые значения env считаются незаданными (`env_ignore_empty` — compose
передаёт `${VAR:-}` как пустую строку).

| Env | Дефолт | Смысл |
|---|---|---|
| `TOAST_DB_HOST/PORT/USER/PASSWORD/NAME` | — | компоненты DSN; не заданы → инструмент неактивен |
| `SQL_MODEL` | `anthropic/claude-sonnet-4.6` | модель generate/judge/summarize |
| `SQL_MAX_QUERIES` | 3 | бюджет SQL-запросов на вызов |
| `SQL_CANDIDATES_PER_ROUND` | 2 | параллельных кандидатов в раунде |
| `LLM_MAX_TOKENS` | см. config.py | предел вывода OpenRouter-моделей (раздел 8) |

## 10. Обвязки

- **`run_sql_tool(inputs, model, executor, …)`** (`toast/sql_tool.py`) —
  низкоуровневый вызов: компилирует граф, гоняет один вход, приводит
  состояние к контракту. Используется тестами и eval.
- **LangGraph Studio** (`studio/`) — dev-раннер графа с формой ввода и
  подсветкой узлов. Собирает тот же граф; `toast/` не знает про Studio.
- **`infra/eval-sql.py`** — диагностический прогон на реальных кейсах
  (must_any/must_not-проверки ответа), не CI-гейт.
- **Демо-режим в чате** (ветка `demo/sql-chat`, см. `docs/sql-demo.md`) —
  профиль «SQL (демо)»: каждое сообщение идёт в граф по демо-таблице,
  стадии видны в UI полным трейсом.
- **Будущая интеграция**: ретривальный пайплайн (пишется отдельно) будет
  находить таблицу под вопрос и вызывать инструмент с её
  `table`/`desc_vector`/`desc_full`. LangChain-обёртка для чат-агента
  намеренно не заведена заранее — появится вместе с реальным контрактом.

## 11. Тестирование

- **Граф** — на фейках (`ScriptedChatModel` — заскриптованные ответы через
  текстовый фолбэк; `StructuredScriptedChatModel` — structured-путь;
  `FakeExecutor` — заскриптованные результаты SELECT). Покрыты: happy path,
  retry по вердикту судьи, исчерпание бюджета, все-ошибки, пустые
  кандидаты, дедупликация, «INSUFFICIENT ≠ sufficient», исключение
  исполнителя, капы контекста.
- **Guardrails** — таблица кейсов на разрешённое/запрещённое, включая
  comma-join, подзапросы, CTE, литералы, закавыченные идентификаторы.
- **Исполнитель** — фейковый `asyncpg.connect`: нет startup-параметров,
  read-only транзакция, LIMIT-обёртка, таймаут/ошибка БД как строки;
  плюс live-тесты (скипаются без `TOAST_DB_*`).
- **Studio** — smoke: граф компилируется, отсутствие env даёт понятную
  ошибку.

## 12. Осознанные ограничения и отложенные решения

- **Рукописный `desc_full` может рассинхронизироваться со схемой** — риск
  принят: имена стабильны, а рассинхрон всплывает Postgres-ошибкой и
  уходит в retry.
- **Вариация temperature между кандидатами** — отложено до сигнала от
  eval, что кандидаты слишком однородны.
- **Поштучный стриминг попыток** (`stream_mode="custom"`) — отложено:
  правки графа ради косметики UI.
- **Счётчики токенов/стоимости** — не делаем без потребности.
- **Судья видит максимум 30 строк / 8 КБ** — для «перечисли всех» на
  больших таблицах вердикт может быть неточен; принято как компромисс
  стоимости.
