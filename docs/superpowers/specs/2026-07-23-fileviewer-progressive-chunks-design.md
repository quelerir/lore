# FileViewer: прогрессивная загрузка чанков + подсветка перехода из чата

Дата: 2026-07-23
Область: `frontend/src/features/files/` (frontend-only, без изменений backend)

## Проблема

При открытии файла с сотнями чанков FileViewer выглядит зависшим: страница ничего
не показывает, пока не догрузит деталь всех чанков.

Корневая причина — `hydrateRunChunks(runId)` (`apiFilesProvider.ts:71`) блокирует
рендер, выполняя три волны запросов **до** возврата результата:

1. Пагинированный обход `/runs/{id}/chunks` — превью всех чанков (без текста).
2. Фан-аут `GET /runs/{id}/chunks/{chunkId}` за полным текстом каждого чанка
   (concurrency 8, до `MAX_CHUNKS = 500`).
3. Батчи payload-refs по 100.

FilesPage ждёт весь `Promise` и только потом рендерит список. На 300+ чанках это
сотни запросов до первого пейнта.

Дополнительно: при клике на цитату в чате пользователь попадает в список из 5–6
чанков и не понимает, на какой именно сослалась LLM — нет ни скролла к чанку, ни
временной подсветки (есть только постоянный `chunkCardActive`).

## Ключевое наблюдение

Центральная панель (список) показывает лишь первую строку `displayText`
(`getChunkCardContent`, `FilesPage.tsx:141`). Полный текст (`displayText/fullText/
vectorText`) нужен только правой панели-инспектору — то есть **одному выбранному**
чанку. Сейчас же деталь тянется для всех 500 ради списка.

`ChunkPreviewDto` (`mappers.ts:93`) текста не содержит — только `chunk_id, ordinal,
chunk_type, content_signature`. Расширять preview-эндпоинт **не будем** (решение:
frontend-only). Значит заголовок карточки берётся из детали, но грузить её нужно
лениво, а не блокирующим фан-аутом.

## Решение (frontend-only)

Разрываем блокирующий метод на два независимых уровня:
**список из превью — сразу**, **деталь — лениво, только для видимого/выбранного**.

### 1. Слой загрузки (`apiFilesProvider.ts`)

Заменяем `hydrateRunChunks` на два метода:

- `listRunChunkPreviews(runId, onPage): Promise<void>` — пагинированный обход
  `/runs/{id}/chunks`, отдающий превью **постранично по мере прихода** через
  `onPage(chunks, { done })`-колбэк (или async-generator). Каждая страница
  маппится через `mapChunkPreview` (плейсхолдер `FileChunk`: ordinal/type/
  signature, текст пустой). Без per-chunk detail, без фан-аута.
- `loadChunkDetail(runId, chunkId): Promise<FileChunk>` — одиночный
  `GET /runs/{id}/chunks/{chunkId}` + резолв его payload-refs. Наполняет
  `displayText/fullText/vectorText/coordinates/payloads` для одного чанка.

Старый `hydrateRunChunks` удаляем; тесты переводим на новые методы. Лимит
`MAX_CHUNKS = 500` сохраняем на уровне превью (превью дешёвые — при желании
поднимается позже, вне скоупа).

### 2. Оркестрация в компоненте (`FilesPage.tsx`)

- **Выбор рана** → `listRunChunkPreviews`. Карточки-плейсхолдеры добавляются в
  состояние пачками по мере страниц → список появляется почти мгновенно.
- **Ленивая деталь через IntersectionObserver.** Наблюдаем за карточками; когда
  чанк входит (или приближается к) вьюпорту → `loadChunkDetail` с ограничением
  параллелизма и дедупом по `chunkId` (статусы `idle | loading | loaded`). По
  приезду детали патчим этот чанк в состоянии → заголовок карточки проявляется.
  На сотнях чанков грузится только реально просматриваемое, а не все 500.
- **Выбранный / deep-link чанк** грузится немедленно и приоритетно, вне
  зависимости от вьюпорта (нужен инспектору справа).
- **Состояние загрузки:**
  - *Хвостовой скелетон* `chunkLoadingRow` — строка-анимация в конце уже
    отрисованного списка, ровно на позиции следующего ещё не приехавшего чанка.
    Рендерится, пока курсор превью не исчерпан (`next_cursor` есть). Естественно
    «съезжает» вниз: нет карточек → он первый; N карточек → (N+1)-й. Убирается,
    когда превью дозагрузились.
  - *Скелетон заголовка* внутри карточки — пока деталь чанка в статусе
    `idle | loading`, вместо заголовка мелкий шиммер; `formatChunkMetaLabel`
    (ordinal · type) показывается сразу.

### 3. Подсветка перехода из чата

- Клик по цитате уже вызывает `navigateTo(deepLink)` → `?chunk=xyz` →
  `selectedChunkId` (через `urlState`).
- Реестр ref'ов карточек: `Map<chunkId, HTMLElement>` через callback-ref на
  `chunkCard`.
- `useEffect` на смену `selectedChunkId`:
  1. гарантирует загрузку детали выбранного чанка (load-on-demand, приоритет);
  2. `scrollIntoView({ behavior: "smooth", block: "center" })` по ref карточки;
  3. вешает временный класс-вспышку `chunkCardFlash` (~1.5–2 c, keyframes фон/
     бордер), снимает по таймеру. Работает поверх постоянного `chunkCardActive`.
- Семантика: `chunkCardActive` = «выбран сейчас», `chunkCardFlash` = «вот сюда
  привела цитата».
- Edge-cases:
  - deep-link пришёл раньше, чем превью с этим чанком дозагрузились → эффект
    ждёт появления целевой карточки в списке (реагирует на её ref/наличие).
  - повторный клик по той же цитате: `navigateTo` ноупает при совпадении пути,
    вспышка не повторится. Минорно; при необходимости лечится nonce в deep-link.

## Компоненты и интерфейсы

| Юнит | Ответственность | Зависит от |
|------|-----------------|-----------|
| `listRunChunkPreviews` | Постраничный поток превью-чанков | `auditGet`, `mapChunkPreview` |
| `loadChunkDetail` | Деталь одного чанка + payloads | `auditGet`, `mapChunkDetail` |
| IntersectionObserver-хук в FilesPage | Триггер ленивой детали для видимого | `loadChunkDetail`, статус-мапа |
| Реестр ref'ов + flash-эффект | Скролл и вспышка для чанка из цитаты | `selectedChunkId`, refs |
| `chunkLoadingRow` (хвост) | Индикатор «список ещё растёт» | флаг незавершённой пагинации превью |
| Скелетон заголовка | Индикатор «карточка ждёт деталь» | статус детали чанка |

## Поток данных

```
selectRun(runId)
  └─ listRunChunkPreviews(runId, onPage)
       └─ onPage(previewChunks, {done}) ──► append placeholders в state
            └─ пока !done: рендерим chunkLoadingRow в хвосте
  IntersectionObserver: card intersects
       └─ статус idle? → loadChunkDetail(runId, chunkId) → patch chunk → заголовок
  citation click → navigateTo → selectedChunkId
       └─ useEffect: ensureDetail(selected) → scrollIntoView → chunkCardFlash (timeout)
```

## Обработка ошибок

- `loadChunkDetail` упал → карточка остаётся плейсхолдером с меткой ordinal·type
  (как сейчас в `catch` внутри `mapPooled`); повтор при следующем пересечении
  вьюпорта (сброс статуса в `idle` на ошибке) или по клику.
- `listRunChunkPreviews` упал на странице → показываем уже приехавшие превью +
  сообщение об ошибке вместо хвостового скелетона.
- Выбор другого рана во время загрузки → отменяем/игнорируем устаревшие ответы
  (guard по актуальному `runId`).

## Тестирование

- `apiFilesProvider.test.ts`:
  - `listRunChunkPreviews` проходит все страницы курсора и **не** дёргает
    detail-эндпоинт; отдаёт страницы через колбэк.
  - `loadChunkDetail` тянет один detail-эндпоинт и резолвит payloads.
  - дедуп/кэш: повторный `loadChunkDetail` того же чанка не шлёт второй запрос.
- Компонентный (FilesPage):
  - плейсхолдер-карточка → заголовок после эмуляции загрузки детали;
  - хвостовой `chunkLoadingRow` виден при незавершённой пагинации, исчезает после;
  - deep-link `?chunk=` → вызывается `scrollIntoView` и вешается `chunkCardFlash`
    (IntersectionObserver и scrollIntoView мокаем).

## Вне скоупа

- Backend/`preview_text` в preview-эндпоинте.
- Поднятие лимита `MAX_CHUNKS = 500`.
- Полноценная виртуализация (react-window) — IntersectionObserver решает проблему
  сотен чанков без неё; вернёмся, если DOM-объём станет узким местом.
- Фоновая догрузка детали всех невидимых чанков (рассмотрено, отклонено в пользу
  ленивой загрузки видимого — меньше трафика, честно масштабируется).
