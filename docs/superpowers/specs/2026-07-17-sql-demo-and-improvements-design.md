# SQL-инструмент: демо хода графа во фронтенде + улучшения

**Дата:** 2026-07-17
**Статус:** утверждён, ждёт плана реализации
**Затрагивает:** `backend/app.py` (+ новый `backend/sql_demo.py`), `backend/config.py`,
`backend/toast/sql_guardrails.py`, `backend/toast/sql_graph.py`,
`frontend/src/chat/*`, `frontend/src/components/{Sidebar,ExecutionSteps}/*`

## Проблема

SQL-инструмент (langgraph-граф в `toast/`) работает и покрыт тестами, но живёт
изолированно: его видно только в LangGraph Studio и eval-скрипте. Реальная
интеграция откладывается — инструменты, которые будут поставлять графу данные
(таблицу и описания), пишутся другим человеком. При этом уже сейчас нужно
показать **дизайн работы инструмента в продуктовом интерфейсе**: как выглядит
ход графа — стадии, вызовы, результаты — в чате, под ответом, со сворачиваемыми
секциями на каждом уровне.

Параллельно принят пакет улучшений самого инструмента (наблюдаемость, качество,
надёжность), не зависящий от будущей интеграции.

## Ограничения среды (жёсткие)

1. **Toast-БД неприкосновенна.** Никаких изменений на стороне БД: ни ролей, ни
   GRANT, ни DDL. Идея read-only юзера с `GRANT SELECT` на `splitter_toast`
   рассмотрена и **отклонена по этой причине**. Защита остаётся целиком на
   стороне приложения: AST-guardrails (sqlglot) + принудительная read-only
   транзакция на каждый запрос. К идее роли не возвращаться.
2. **Совместимость со Studio.** Сигнатура `build_sql_graph` и вход/выход графа
   не меняются; в `toast/` не появляется ни одного импорта Chainlit;
   `studio/graph.py` не трогается; smoke-тесты Studio остаются зелёными.
3. **Демо — в отдельной ветке.** Улучшения графа (разделы 4.x) идут в основную
   ветку `toast-logic`; демо (профиль, обработчик, фронтенд) — в ветке
   `demo/sql-chat` поверх неё. Демо-ветка не предназначена для merge в main.

## Решение (принятое) и отклонённые альтернативы

Ход графа доставляется во фронтенд через **`cl.Step` на каждую стадию**,
порождаемые из `graph.astream(..., stream_mode=["updates", "messages"])` в слое
Chainlit (подход A). Шаги уходят по вебсокету живьём и persist'ятся data
layer'ом — открытие старого треда воспроизводит демонстрацию.

Отклонено:
- **Автошаги `LangchainCallbackHandler`** — имена и вложенность шагов не
  контролируются, chain-шаги скрыты конфигом; для демонстрации дизайна
  неуправляемо.
- **Свой JSON-протокол / кастомные элементы** — одноразовый код мимо
  существующей инфраструктуры шагов; оправдан только для отдельной страницы.
- **Отдельная демо-страница** — дальше от продуктового вида; выбран чат.
- **Триггер через агента (`query_table` у fast-агента)** — недетерминированно
  для показа; выбран отдельный чат-профиль, где каждое сообщение идёт в граф.

## Дизайн

### 1. Демо-режим бэкенда (ветка `demo/sql-chat`)

**Профиль.** Третий чат-профиль `sql` («SQL (демо)») в `chat_profiles()`.
Регистрируется **только** при сконфигурированных `toast_dsn` и
`OPENROUTER_API_KEY`; без кредов профиля нет — полурабочих состояний не бывает.

**Сессия.** `on_chat_start` для `sql` кладёт в сессию скомпилированный граф
(`build_sql_graph` + `PgExecutor(toast_dsn)` + `build_sql_model()`), а не
агента. Граф без памяти диалога: каждое сообщение — независимый прогон.
`on_chat_resume` ничего не восстанавливает (история сообщений и шагов
persist'ится data layer'ом сама).

**Демо-таблица.** Дефолты захардкожены из eval-кейса «юристы»
(`toast_tbl_ec48a6d52d16ab405f95` + desc_vector/desc_full), переопределяются
через env `SQL_DEMO_TABLE` / `SQL_DEMO_DESC_VECTOR` / `SQL_DEMO_DESC_FULL`
(поля в `config.Settings` рядом с остальными `SQL_*`) — таблицу можно сменить
на показе без правки кода.

**Обработчик.** `on_message` при `sql`-режиме уходит в `handle_sql_message()`
(отдельный модуль `backend/sql_demo.py`, чтобы не разбухал `app.py`):

- события `updates` (узел завершился) → `cl.Step`:
  - `generate` → «Генерация SQL — раунд N», output = кандидаты;
  - `execute` → «Выполнение SQL — раунд N», внутри вложенный под-шаг на каждую
    попытку: input = SQL, output = превью строк (первые ~5 + `row_count`) или
    текст ошибки; `isError` для неуспешных;
  - `judge` → «Оценка достаточности», output = вердикт;
  - `init` шаг не создаёт;
- события `messages` → токены `summarize` (единственный незатегированный
  LLM-вызов) стримятся в сообщение ответа, как в `handle_message`.

Конвертация `(узел, дельта состояния) → (имя, input/output, под-шаги)` — чистая
функция, тестируется без Chainlit и без графа.

**Принятое ограничение:** под-шаги попыток появляются разом по завершении узла
`execute` (попытки бегут параллельно внутри узла; поштучный стриминг потребовал
бы `stream_mode="custom"` и правок графа — отклонено, см. «Не делаем»).

### 2. Фронтенд (ветка `demo/sql-chat`)

**Режим.** `ChatMode = "fast" | "deep" | "sql"`, третья кнопка «SQL (демо)» в
`Sidebar`. Существующий `handleModeChange` уже начинает новый чат при смене
режима — этого достаточно. Кнопка статическая; фильтрацию по списку профилей с
сервера не делаем (ветка живёт только в сконфигурированном окружении).

**Сбор шагов** (`executionSteps.ts`). `gatherTools` сейчас сплющивает все
tool-шаги поддерева. Новый контракт: собираются только **верхнеуровневые**
tool-шаги (спуск останавливается на первом `type === "tool"`), дети остаются
внутри через `step.steps`. Для fast/deep поведение визуально не меняется
(у калькулятора нет вложенных шагов).

**Рендер** (`ExecutionSteps.tsx`). Рекурсивный `StepItem`:
- каждый шаг — `<details>`/`<summary>`: имя + маркер статуса
  (✓ / ✗ по `isError` / «…» пока `streaming`);
- внутри: input (моноширинный `<pre>` — SQL), output (превью строк / вердикт /
  кандидаты), вложенный список `step.steps` тем же компонентом;
- корневая панель «Ход выполнения» как есть (`open={running}`); вложенные
  стадии по умолчанию свёрнуты, кроме выполняющейся. Сворачиваемость — на
  каждом уровне.

Стили в существующем `ExecutionSteps.module.css`, без новых зависимостей.
`threadToChat`/resume не меняются: дерево шагов из data layer имеет ту же
вложенность, что и по вебсокету.

### 3. Улучшения графа (ветка `toast-logic`)

**3.1 Разрешить CTE** (`sql_guardrails.py`). Собрать алиасы CTE
(`stmt.find_all(exp.CTE)` → `alias_or_name`); при проверке таблиц пропускать
ссылки **без схемы**, чьё имя есть среди алиасов. Таблица со схемой внутри CTE
(`WITH x AS (SELECT * FROM public.users) …`) — по-прежнему отказ; среди
relation обязана остаться хотя бы одна настоящая `splitter_toast.<table>`.
Отказ «CTE не поддерживается» удаляется; из `GENERATE_SYS` уходит запрет WITH.

**3.2 Structured output с фолбэком** (`sql_graph.py`). `generate` сначала
пробует `model.with_structured_output(SqlCandidates, method="function_calling")`
(pydantic: `{candidates: list[str]}`, обрезка до `n`). Любое исключение
(OpenRouter-модель без function calling, `NotImplementedError` фейка в тестах)
→ фолбэк на `ainvoke` + `parse_sql_candidates`. Парсер не удаляется — он и
есть фолбэк. Тег `internal` сохраняется в обоих путях.

**3.3 Кап контекста судьи по размеру** (`sql_graph.py`, `_rows_context`).
К лимиту 30 строк добавляется лимит сериализованного JSON (~8 000 символов):
строки добавляются, пока влезают. Пометка «Показано строк: X из Y» остаётся
честной.

### 4. Обработка ошибок

- `status=error` / `no_data` от графа — текст в сообщении (граф уже делает);
  неуспешные попытки — красные под-шаги (`isError`).
- Исключение прогона (сеть до OpenRouter и т.п.) — ловится в
  `handle_sql_message`, пользователю короткое сообщение об ошибке, стек в лог.
- Нет кредов — профиль не регистрируется (см. раздел 1).

### 5. Тестирование

Бэкенд:
- чистая функция конвертации `(узел, дельта) → шаг`: generate/execute/judge,
  ошибки попыток, нумерация раундов — без Chainlit и графа;
- условная регистрация профиля `sql`;
- guardrails: CTE по своей таблице проходит, CTE поверх чужой — отказ, CTE без
  единой реальной таблицы — отказ «не вижу FROM»;
- structured output: основной путь + фолбэк по исключению;
- кап контекста судьи: гигантская строка не раздувает контекст.

Фронтенд:
- `executionSteps.test.ts`: иерархия вместо плоского списка (вложенные шаги
  остаются детьми, верхний уровень их не дублирует);
- рендер-тест `ExecutionSteps` с двухуровневым деревом (vitest + happy-dom).

Регресс: smoke-тесты Studio; ручной прогон демо фиксируется в README
демо-ветки.

## Не делаем (осознанно)

- **Роль/GRANT в toast-БД** — запрещено ограничением среды (БД неприкосновенна).
- **Вариация temperature между кандидатами** — ждём сигнала от eval, что
  кандидаты слишком однородны.
- **Поштучный стриминг попыток** (`stream_mode="custom"`) — правки графа ради
  косметики демо.
- **Чтение списка профилей с сервера во фронтенде** — демо-ветка живёт в
  сконфигурированном окружении.
- **`query_table` у fast-агента** — это часть настоящей интеграции, не демо.
