# SQL-инструмент: фиксы по внешнему ревью схемы

**Дата:** 2026-07-17
**Статус:** утверждён, ждёт плана реализации
**Ветка:** `toast-logic` (демо получит через merge)
**Источник:** `docs/sql-tool-review.md` — внешнее ревью `docs/sql-tool.md`,
ключевые претензии воспроизведены на живом коде (обход guardrails через
`query_to_xml` и слом многострочного SQL в фолбэке — подтверждены).
**Затрагивает:** `backend/toast/sql_graph.py`, `sql_guardrails.py`,
`executor.py`, тесты, `docs/sql-tool.md`.

## Проблема

Ревью нашло: (а) дыру безопасности — guardrails не проверяют вызовы функций,
а Postgres имеет штатные функции, исполняющие SQL-текст из аргумента
(`query_to_xml`), read-only транзакция от такого ЧТЕНИЯ не защищает;
(б) сломанный retry-цикл — вердикт судьи не попадает в фидбек generate, при
temperature 0 новый раунд порождает те же кандидаты, дедупликация их
отсеивает, бюджет сгорает без единого нового SQL; (в) неиспользованный рычаг
качества — модель пишет SQL вслепую относительно реальных значений в данных;
плюс набор подтверждённой гигиены.

## Принятые решения

### 1. Денайлист функций в guardrails (ревью 2.1) — безопасность

В `validate_select` добавляется проверка всех вызовов функций в AST
(`exp.Anonymous` и типизированные функции sqlglot) против денайлиста по
имени (без учёта регистра): `query_to_xml*`, `xmltable`, `dblink*`,
`pg_read_*`, `pg_ls_*`, `pg_stat_file`, `lo_import`, `lo_export`,
`pg_sleep*`, `pg_terminate_backend`, `pg_cancel_backend`, `current_setting`,
`set_config`. Агрегаты и обычные скалярные функции остаются разрешёнными.
Отказ: «Отказ: функция <имя> запрещена.»

Денайлист (а не аллоулист) — осознанно: аллоулист всех легитимных функций
Postgres нереалистично поддерживать, а классы опасных функций известны и
конечны. В `docs/sql-tool.md` формулировка «слои независимы — пробитие
одного не открывает данные» заменяется честной: слой 2 защищает только от
записи, конфиденциальность чужих схем держится на слое 1.

### 2. Судья с причиной (ревью 2.2) — замыкание retry-цикла

Судья переводится на structured output `JudgeVerdict {sufficient: bool,
reason: str}` (method="function_calling") с фолбэком на текущий текстовый
парсинг (граница слова + приоритет `need_more`), reason при фолбэке пуст.
`reason` кладётся в состояние (`judge_reason`) и добавляется в промпт
следующего generate отдельной секцией: «Прошлый результат отклонён судьёй:
<reason> — построй запрос иначе». Симметрично существующему фидбеку ошибок
и пустых запросов.

### 3. Бюджет и предел раундов (ревью 2.4)

`executed_count` увеличивается ТОЛЬКО на число реально выполненных SQL
(после дедупликации и отказов guardrails): отказ валидатора не съедает
бюджет — его фидбек существует, чтобы модель переписала запрос. Гарантия
завершаемости переносится с «батч всегда списывает ≥1» на явный предел
раундов: `round >= max_queries` в `after_execute` → summarize. Семантика
из `docs/sql-tool.md` («предел числа ВЫПОЛНЕННЫХ SQL») становится правдой.

### 4. Состояние на pydantic; узел `init` удаляется

Состояние графа переводится с `TypedDict(total=False)` на pydantic-модель с
дефолтами полей (`attempts=[]`, `executed_count=0`, `round=0`, …) — langgraph
применяет дефолты сам, и единственная работа узла `init` (гарантированное
создание аккумуляторов) исчезает вместе с узлом. Причина существования
`init` была ровно в том, что у TypedDict-состояния нет дефолтов (инцидент
`KeyError: 'candidates'`); pydantic закрывает этот класс ошибок на уровне
схемы. Узлы переходят с `state["поле"]` на `state.поле`; `SqlToolInput`
остаётся схемой входа; выход `ainvoke` — как прежде, словарь значений
(контракт `_project` не меняется).

Топология после правок: `START → sample → generate → execute → judge →
summarize → END` (условные переходы прежние).

### 5. Узел `sample` — примеры строк в промпте (ревью 1.1)

Новый ОТДЕЛЬНЫЙ узел графа, ПЕРВЫЙ в топологии (по явному решению — для
наглядности работы графа в Studio и трейсе):

- выполняет фиксированный `SELECT * FROM splitter_toast.<table> LIMIT 5`
  через тот же `executor.run_select` (guardrails его пропускают штатно);
- НЕ списывает бюджет (детерминированный дешёвый запрос);
- кладёт строки в `state.sample_rows`; сбой не фатален: пустой список +
  warning в лог, граф продолжает;
- generate добавляет в промпт секцию «Примеры строк (до 5):» с JSON строк,
  усечённым по размеру (~2000 символов) — модель видит реальные имена
  колонок и формат значений;
- вход/выход графа НЕ меняются (Studio-совместимость сохранена); в
  топологии появляется узел — это желаемая видимость.

Побочный эффект: рассинхрон `desc_full` со схемой всплывает до траты
бюджета. Тесты графа получают один дополнительный заскриптованный результат
в `FakeExecutor` (первый `run_select` каждого прогона — сэмпл).

### 6. Гигиена (ревью 2.5–2.8)

- **2.5**: текстовый фолбэк `parse_sql_candidates` перестаёт быть
  построчным — не-JSON текст парсится `sqlglot.parse`, кандидаты — это
  распарсенные SELECT-стейтменты (многострочный SQL цел); при полном
  провале парсинга — прежнее поведение для строки, начинающейся с SELECT.
- **2.6**: `qualify_table` переводится с regex на AST-трансформ sqlglot
  (голое имя таблицы = переданной → проставить схему); литералы больше
  недостижимы. При неразбираемом SQL — вернуть текст как есть (упадёт в
  validate с внятным отказом).
- **2.7**: `asyncpg.connect(..., statement_cache_size=0)` — обязательное
  ограничение для transaction-pooling пулера; фиксируется в
  `docs/sql-tool.md` раздел 2.
- **2.8**: в `_generate_candidates` фолбэк логирует причину
  (`logging.warning` с типом исключения); `NotImplementedError` (фейки,
  модели без tools) — `debug`, остальное — `warning`.

### 7. Лёгкие версии (ревью 1.3, 2.3)

- **1.3-lite**: `_rows_context` группирует строки по попыткам с их SQL
  («Запрос: <sql>\nСтроки: [...]») вместо слепого объединения — судья и
  суммаризатор видят, какой запрос что вернул. Капы (30 строк / 8 КБ
  суммарно) сохраняются.
- **2.3-lite**: промпт summarize дополняется требованием явно сообщать в
  ответе о неполноте выборки, если показаны не все строки; в
  `docs/sql-tool.md` кап на финальном ответе признаётся открыто;
  `rows_used` в контракте документируется как «строк получено успешными
  попытками» (не переименовывается — поле уже используется обвязками).

## Не делаем (осознанно)

- Execution-based selection вместо судьи (перестройка узла без данных о
  необходимости).
- «Льготные» exploration-запросы модели (sample покрывает основную нужду).
- Дешёвая модель для judge/summarize (усложнение конфига до измерений).
- Поднятие капа строк для summarize (ждём сигнала с ручных прогонов).
- Детерминированная валидация колонок в guardrails (модель видит реальные
  колонки из sample; связывать guardrails со схемой БД не хотим).

## Ограничения (без изменений)

Toast-БД неприкосновенна; вход/выход графа и сигнатура `build_sql_graph`
стабильны; `toast/` без импортов Chainlit; smoke-тесты Studio зелёные.

## Тестирование

- guardrails: `query_to_xml`/`xmltable`/`dblink`/`pg_sleep` — отказ;
  `count(*)`, `lower()`, `coalesce()` — проходят; qualify_table:
  литерал с подстрокой `from toast_tbl_…` не переписывается.
- граф: sample-узел в топологии (первый run_select — сэмпл, не списывает
  бюджет; сбой сэмпла не фатален); узла init нет, дефолты состояния — от
  pydantic-схемы; отказ guardrails не съедает бюджет; предел раундов
  останавливает цикл при батчах из одних отказов; reason судьи доезжает до
  промпта generate (проверка через записанный промпт фейка); группировка
  _rows_context по попыткам.
- парсер: многострочный SELECT в не-JSON фолбэке разбирается целиком.
- executor: `statement_cache_size=0` присутствует в kwargs connect.
- Studio: существующие smoke-тесты.
