# SQL-инструмент: убрать зависимость `scope` от БД, довериться `desc_full`

**Дата:** 2026-07-17
**Статус:** утверждён, ждёт плана реализации
**Затрагивает:** `backend/toast/sql_graph.py`, `studio/test_graph_smoke.py`, тесты графа

## Проблема

Модель генерирует SQL с несуществующими колонками (напр. `SELECT col_5, col_4 …`, где `col_5` нет). Разбор показал две разные причины:

1. **Дисциплина**: модель придумывает колонку, которой нет в списке (разреженная нумерация `col_4, col_8` без `col_5`). Слабая модель (бесплатный nemotron) усугубляет.
2. **Семантический мост**: физические имена колонок непрозрачны (`col_4`, `col_8`), а `desc_full` описывает смысл по-человечески («Реестр юристов: ФИО, должность, email») без привязки к физическим колонкам. Модели неоткуда узнать, какая колонка что значит.

Ключевое наблюдение: **`desc_full` будет содержать готовый маппинг** `col_4 → ФИО`, `col_8 → должность` с реальными физическими именами (вариант A). Описание пишется **человеком** и передаётся **не через LLM**, поэтому считается авторитетным источником имён и смысла колонок.

## Решение (принятое)

Полностью довериться `desc_full` как источнику имён+смысла колонок и **убрать обращение узла `scope` к БД** (`fetch_columns`). Узел `scope` тянул реальные колонки из `information_schema` — при наличии авторитетного человеческого описания это дублирование, причём беднее (имена без смысла).

Рассмотренная альтернатива (оставить `scope` как независимый источник истины для детерминированной валидации колонок) отклонена: описание рукописное и стабильное, а единственный остаточный риск (rename/drop физической колонки без обновления описания) уже закрыт существующим бэкстопом — Postgres-ошибкой на выполнении, которая уходит в retry-цикл.

### Честная оговорка (проговорена и принята)

«Человеческое» описание ≠ «синхронное со схемой». При физическом переименовании колонки (`col_4 → col_9`) рукописный `desc_full` останется устаревшим, пока человек вручную его не поправит. Этот риск **осознанно принят**, потому что:
- физические имена колонок в системе стабильны;
- при рассинхроне SQL упадёт на выполнении, и Postgres-ошибка (`column col_4 does not exist`) уйдёт в `errors` → `generate` → retry. Система не летит вслепую даже без `scope`.

## Дизайн

### Топология графа

```
START → init → generate → execute(∥) → judge ─┐
                  ▲                      │      │
    (нужен раунд) └──────────< generate ─┘      │
                                                ▼
                       (бюджет исчерпан) → summarize → END
```

Единственное изменение против текущего графа: узел `scope` (ходил в БД) → узел `init` (без БД). Маршрутизация (`after_execute`, `after_judge`) и retry-цикл — без изменений.

### Узлы

1. **`init`** — детерминированный, без БД. Возвращает `{"attempts": [], "executed_count": 0, "round": 0}`. Единственная ответственность — инициализация аккумуляторов, чтобы первый `execute`/`generate` не падал на отсутствующем ключе (тот же класс ошибки, что недавний `KeyError: 'candidates'`). Явный узел выбран сознательно вместо размазанных `state.get(...)` дефолтов — централизация защищает от footgun'а с забытым ключом.

2. **`generate`** — источником имён+смысла колонок становится `desc_full` (он уже в промпте строкой `Описание (полно): {desc_full}`). Правки промпта:
   - убрать строку `Реальные колонки: {', '.join(state['columns'])}`;
   - в константе `FIXED_SCHEMA` заменить хвост «Используй ТОЛЬКО реальные имена колонок из списка ниже» → «Используй физические имена колонок строго как в описании таблицы». Общая информация (служебные колонки `_splitter_*`, конвенция именования `column_N`) остаётся.

3. **`execute` / `judge` / `summarize`** — без изменений.

### Чистка состояния

- Из `SqlToolState` убрать поле `columns` — под новой топологией его никто не пишет и не читает.
- Обновить docstring графа: диаграмму топологии и описание узлов (`scope` → `init`), убрать упоминание добычи колонок из БД.

### Executor

- `executor.fetch_columns` **оставляем** в классе `PgExecutor`. Это легитимная возможность исполнителя, покрыта отдельным тестом (`test_fetch_columns_includes_service_and_renamed`), может пригодиться под будущий guardrail. Граф её просто больше не вызывает. Мёртвым кодом не считаем — churn не оправдан.

### Коррекция рассинхрона

Отдельного кода не требует. При устаревшем `desc_full` SQL падает на выполнении → Postgres-ошибка → `errors = [a["error"] for a in state["attempts"] if not a["ok"]]` в `generate` → следующий раунд retry с подсказкой БД.

## Вне области

- Детерминированный guardrail на колонки (валидация SQL против реального списка колонок до выполнения) — отклонён в пользу доверия `desc_full` + бэкстопа на выполнении.
- Подмешивание сэмпла строк в контекст — не требуется при варианте A (`desc_full` несёт маппинг).
- `Annotated`-редьюсеры (`operator.add`) для `attempts` — больший рефактор с изменением семантики мёржа, не оправдан.
- Смена модели — рассматривается отдельно и только после проверки нового контекста на реальных вопросах.
