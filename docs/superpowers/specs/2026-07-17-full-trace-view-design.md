# Полный трейс работы в чате (LangSmith-подобный список)

**Дата:** 2026-07-17
**Статус:** утверждён, ждёт плана реализации
**Ветка:** `demo/sql-chat` (поверх рекурсивного рендера шагов)
**Затрагивает:** `frontend/src/chat/executionSteps.ts`,
`frontend/src/components/ExecutionSteps/*`, `backend/sql_demo.py`

## Проблема

Панель «Ход выполнения» показывает только `tool`-шаги. Для дебага нужно видеть
ВСЕ этапы работы Chainlit — LLM-вызовы с промптами/ответами, tool-вызовы,
run-контейнеры — деревом, приближенно к списку трейса LangSmith. Существующий
сворачиваемый рендер делает это удобным: узлы по умолчанию свёрнуты и не мешают
обычному пользователю.

## Решения (приняты)

- **Всегда включено, та же панель.** Никаких тумблеров и env-флагов: полный
  трейс виден всем в «Ходе выполнения». Отклонены: debug-переключатель в UI,
  сборочный флаг, отдельная страница-инспектор.
- **Состав:** все типы шагов (llm/tool/run) + длительность каждого шага.
  Счётчики токенов НЕ делаем (Chainlit-шаги usage не несут, прокидывать
  через metadata — отдельная работа без текущей потребности).
- **SQL-режим тоже трассируется:** сейчас `handle_sql_message` не передаёт
  колбэк, и LLM-вызовы графа в шаги не попадают вовсе.

Фактическая база: `cot` в Chainlit не переопределён → дефолт `"full"`, сервер
уже шлёт клиенту все шаги (llm/tool/run с input/output и start/end). Менять
серверную фильтрацию не нужно.

## Дизайн

### 1. Сбор (`executionSteps.ts`)

`collectToolStepsByMessage` переименовывается в `collectTraceByMessage`
(сигнатура та же: `(steps: IStep[]) => Map<string, IStep[]>`). Логика: для
каждого run'а, среди прямых детей которого есть `assistant_message`, в Map
кладутся ВСЕ его дети-шаги, кроме шагов с типами `*_message` (user/assistant
message — это контент чата, не трейс). Вложенность `step.steps` не трогается —
никакого сплющивания и никакой фильтрации по типу: llm-шаги и run-контейнеры
входят в дерево.

Единственный потребитель — `ChainlitRuntimeProvider` (контекст
`sessionUi.toolStepsByMessage` → переименовать в `traceByMessage`) и
`AssistantMessage`.

### 2. Рендер (`ExecutionSteps.tsx` + module.css)

`StepItem` дополняется:
- бейдж типа слева от имени: `llm` / `tool` / `run` (моноширинный, приглушённый);
- длительность справа: `end − start`; < 1 c → «N мс», иначе «N.N с»;
  формат — чистая функция `formatDuration(startMs, endMs)`;
- незавершённый шаг — «…» (как сейчас), `isError` — «✗» (как сейчас);
- `<pre>` для input/output рендерится только если строка непуста (у
  run-контейнеров их обычно нет);
- счётчик в шапке панели — число верхнеуровневых узлов трейса.

Сворачиваемость на каждом уровне, авто-раскрытие выполняющегося узла и
`max-height` + скролл у `<pre>` уже есть — не меняются.

### 3. Бэкенд (`sql_demo.py`)

В `graph.astream(...)` добавляется `config` с `cl.LangchainCallbackHandler()` —
LLM-вызовы generate/judge/summarize становятся llm-шагами с промптами и
ответами. Тег `internal` продолжает скрывать их ТОКЕНЫ из стрима сообщения;
сами шаги — видимы, это и есть цель. Рукотворные стадии-`cl.Step` остаются;
llm-шаги ложатся рядом в дереве (порядок событий может отличаться от порядка
стадий — принято, это дебаг-вью). fast/deep не меняются: колбэк там уже стоит.

### 4. Крайние случаи

- Резюм старых тредов: дерево шагов из data layer той же формы — работает само.
- Огромные промпты: `<pre>` со скроллом, узлы свёрнуты по умолчанию.
- Шаг без `start`/`end` (не завершён/потерян): длительность не показываем.

### 5. Тестирование

Фронтенд (vitest):
- сборщик: llm-шаги и run-контейнеры входят, `*_message` исключены,
  вложенность сохранена; существующие кейсы переименованного сборщика;
- `formatDuration`: мс / секунды / отсутствующие границы;
- рендер: бейджи типов и длительность в двухуровневом дереве.

Бэкенд: изменений тестов нет (колбэк — параметр вызова; проверяется ручным
прогоном демо). Регресс: `npm test`, `npm run build`, backend pytest.

## Не делаем (осознанно)

- Фильтры/поиск по трейсу, копирование JSON шага.
