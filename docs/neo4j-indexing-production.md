Статус: **бэклог, не срочно.** Дата: 2026-07-22.

Индексация корпуса в Neo4j (dense-эмбеддинги + fulltext + структурный граф) —
это офлайн-джоба, отдельная от request-пути (чат только **читает** Neo4j).
Сейчас она работает, но как **spike-скрипт, запускаемый вручную**. Ниже — что
нужно, чтобы довести её до прод-зрелости. Опирается на раздел «Indexing
Pipeline» / «Derived-index ledger» спеки `lore-neo4j-native-...-draft.md`.

## Где мы сейчас
- `lore-core/packages/lore-retrieval/spikes/index_corpus.py` — читает
  `lore_core.chunks` (Postgres) → эмбедит `vector_text` через Ollama/bge-m3 →
  пишет узлы `TextChunk_vN`/`TableChunk_vN` + `Section_vN` + индексы
  `vec_*`/`ft_*` в Neo4j (`MERGE` по `projection_id`, идемпотентно).
- Версия берётся из `index_version` (конфиг). Чат читает ровно одну версию.
- Запуск — руками: `RETRIEVAL_INDEX_VERSION=vN ... uv run python spikes/index_corpus.py --limit N`.

## Что нужно для прода

### 🔴 P1. Вынести из `spikes/` в настоящую джобу
- CLI-энтрипоинт пакета (`lore_retrieval.index` / `python -m lore_retrieval.index`),
  а не скрипт в `spikes/`.
- Контейнеризовать как отдельную задачу (k8s Job / cron-контейнер / compose
  `profiles: [indexing]`) с тем же общим `.env`, что и чат.
- Без хардкод `--limit`: индексировать весь готовый прогон целиком.

### 🔴 P2. Derived-index ledger (реестр версий)
Таблица (Postgres) со статусами проекций: `run_id, index_version,
embedding_model_version, fulltext_analyzer_version, graph_schema_version,
neo4j_version, status(pending|indexing|ready|failed|superseded),
started_at, completed_at, error_summary`. Джоба пишет свой прогресс сюда.

### 🔴 P3. Атомарная активация «одной готовой версии»
- Строить новую версию (vN+1) **рядом** с текущей (не перезаписывая активную).
- Переключать чат на неё **только после** `status=ready` + прохождения
  retrieval-проверки. Активную версию хранить как источник для чтения;
  предыдущую готовую — как откат.
- Механика переключения: одно значение (`index_version`) в общем `.env` /
  конфиге + перезапуск/пересборка читателей. Автоматизировать в джобе.

### 🟡 P4. Очистка устаревших проекций
После активации новой версии — пометить старую `superseded` и удалять по
retention-политике (`DROP INDEX vec_*_vOLD` + `MATCH (n:*_vOLD) DETACH DELETE`).
Сейчас старые узлы остаются висеть (v1 с прошлым корпусом — живой пример).

### 🟡 P5. Триггер после обработки корпуса
Джоба должна запускаться **по событию** завершения обработки нового прогона в
lore_core (хук/очередь/расписание), а не вручную. До этого — задокументировать
ручной ранбук.

### 🟡 P6. Наблюдаемость и устойчивость
- Прогресс-логи по батчам (сейчас stdout буферизован, прогресса не видно).
- Ретраи на транзиентных обрывах Neo4j/Ollama (VPN флапает — джоба на 19k
  чанков не должна падать целиком).
- Метрики: число узлов, длительность, стоимость эмбеддинга.

### ⚪ P7. Инкрементальная индексация
Сейчас переиндексируется весь корпус. При больших корпусах — индексировать
только изменённые/новые чанки (diff по `fulltext_hash`).

## Связанные факты
- Источник истины — Postgres (`lore_core.chunks` + `splitter_toast.toast_tbl_*`);
  Neo4j — производная пересобираемая проекция, никогда не единственная копия.
- Физические таблицы для SQL лежат в схеме `splitter_toast` (см.
  `toast/sql_guardrails.py: ALLOWED_SCHEMA`).
